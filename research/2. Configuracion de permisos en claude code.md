# Configuracion de Permisos en Claude Code: Guia Practica

Claude Code pide aprobacion manual cada vez que ejecuta un comando potencialmente peligroso. Esto es bueno para seguridad, pero cuando estas en flow y te interrumpe por un `git status` o un `ls`, se vuelve frustrante rapido.

La solucion es configurar un **allowlist** de permisos para los comandos que sabes que son seguros.

## Niveles de Permisos

Claude Code maneja permisos en una jerarquia donde cada nivel superior sobreescribe al inferior:

**Por sesion** -- Usando el comando `/permissions` dentro de la sesion activa. Solo aplica a esa sesion y se pierde al cerrarla.

**Por proyecto** -- Editando `.claude/settings.json` en la raiz del repositorio. Aplica a cualquier sesion de Claude Code iniciada en ese proyecto.

**Global (usuario)** -- Editando `~/.claude/settings.json`. Aplica a todos los proyectos del usuario.

**Enterprise** -- Archivo `managed-settings.json` a nivel sistema. Tiene prioridad sobre todo lo demas y es gestionado por administradores.

## Sintaxis

Los permisos se definen en el bloque `permissions` del `settings.json` con dos arrays: `allow` y `deny`.

```json
{
  "permissions": {
    "allow": [
      "Bash(comando:*)"
    ],
    "deny": []
  }
}
```

El patron `Bash(comando:*)` permite cualquier invocacion de ese comando con cualquier argumento. Por ejemplo, `Bash(git:*)` permite `git commit`, `git push`, `git status`, etc.

## Permisos para MCP Servers

Si usas MCP servers, la sintaxis es diferente. Para permitir **todas** las herramientas de un MCP server:

```json
"mcp__nombre_del_server"
```

Para permitir una herramienta especifica de un MCP server:

```json
"mcp__nombre_del_server__nombre_de_tool"
```

Por ejemplo, `mcp__vibeMCP` permite todas las tools del server vibeMCP.

## Configuracion Recomendada

Esta configuracion balancea productividad y seguridad. Permite operaciones cotidianas de desarrollo sin friccion, manteniendo aprobacion manual para operaciones destructivas.

```json
{
  "permissions": {
    "allow": [
      "Bash(git:*)",
      "Bash(gh:*)",
      "Bash(swift:*)",
      "Bash(xcodebuild:*)",
      "Bash(xcrun:*)",
      "Bash(python:*)",
      "Bash(pip:*)",
      "Bash(uv:*)",
      "Bash(docker:*)",
      "Bash(npm:*)",
      "Bash(npx:*)",
      "Bash(cat:*)",
      "Bash(ls:*)",
      "Bash(find:*)",
      "Bash(grep:*)",
      "Bash(head:*)",
      "Bash(tail:*)",
      "Bash(wc:*)",
      "Bash(jq:*)",
      "Bash(mkdir:*)",
      "Bash(cp:*)",
      "Bash(mv:*)",
      "Bash(touch:*)",
      "Bash(curl:*)"
    ],
    "deny": []
  }
}
```

### Desglose por categoria

**Git y GitHub CLI** -- `git` y `gh` para operaciones de control de versiones y gestion de PRs/issues.

**iOS / Swift** -- `swift` (build, test, package), `xcodebuild` (compilacion y tests de Xcode), `xcrun` (simuladores con simctl y otras herramientas de Xcode).

**Python** -- `python` para scripts y tests, `pip` para instalacion de paquetes, `uv` como package manager alternativo.

**Docker** -- Para manejo de contenedores, compose, logs y builds.

**Node** -- `npm` y `npx` para tooling y ejecucion de MCP servers.

**Lectura y exploracion** -- `cat`, `ls`, `find`, `grep`, `head`, `tail`, `wc`, `jq`. Operaciones de solo lectura que nunca deberian requerir aprobacion.

**File management** -- `mkdir`, `cp`, `mv`, `touch`. Creacion y movimiento de archivos.

**Network** -- `curl` para testing de APIs.

## Lo que NO deberia estar en el allowlist

Estos comandos deben seguir pidiendo aprobacion manual:

- `rm` -- Eliminacion de archivos. Siempre queres ver que va a borrar.
- `sudo` -- Ejecucion con privilegios elevados.
- `chmod` / `chown` -- Cambio de permisos y ownership de archivos.
- `kill` / `killall` -- Terminacion de procesos.
- Cualquier comando que modifique configuracion del sistema.

## Donde aplicar

Si un permiso es relevante para todos tus proyectos (como git, lectura de archivos, o un MCP server que usas siempre), ponelo en el **global**: `~/.claude/settings.json`.

Si es especifico a un proyecto (como un MCP server particular o un framework que solo usas ahi), ponelo en el **proyecto**: `.claude/settings.json` en la raiz del repo.

La regla es simple: si lo necesitas en mas de un proyecto, va en global.

## Gestion incremental

No intentes configurar todo de entrada. Empieza con lo minimo y anda agregando permisos a medida que las interrupciones se vuelven molestas. Cuando Claude Code te pida permiso para algo que haces seguido, ese es el momento de agregarlo al allowlist.

Podes hacerlo rapido desde la sesion con `/permissions` y despues migrar la regla al `settings.json` cuando confirmes que la queres permanente.
