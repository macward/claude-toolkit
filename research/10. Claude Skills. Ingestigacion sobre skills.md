# Resultados

El proyecto sin skills estuvo mucho mejor diagramado y ejecutado.
El resultado es contraintuitivo pero tiene sentido. Analizemos:
## El problema: Las skills enseñan el "qué" pero no el "nivel de calidad"

| Lo que las skills dicen | Lo que faltó                                     |
| ----------------------- | ------------------------------------------------ |
| "Usá `$props()`"        | "Usá TypeScript nativo, no JSDoc"                |
| "Usá callback props"    | "Pasá solo IDs, no objetos completos"            |
| "Usá `$derived`"        | "Extraé clases complejas a `$derived`"           |
| "Accesibilidad"         | "Focus rings, aspect ratios, cleanup en effects" |

## Por qué el proyecto sin skills fue mejor

Sin restricciones, Claude usó su **conocimiento general completo**:

- Eligió TypeScript nativo (mejor práctica general)
- Usó lucide-svelte (conocimiento de ecosistema)
- Agregó Storybook completo (experiencia en proyectos reales)
- Hizo cleanup en `$effect` (conocimiento profundo de Svelte)

Con skills, Claude se **limitó a seguir las reglas** sin ir más allá.

---

## Cómo mejorar las skills

Las reglas actuales son **descriptivas** (qué hacer), deberían ser **prescriptivas con estándares** (qué hacer + nivel de calidad).

**Antes:**
```markdown
## Use $props()
Use `$props()` para declarar props.
```

**Después:**

```markdown
## Use $props() with TypeScript
SIEMPRE usar TypeScript nativo con interfaces, nunca JSDoc.

**❌ Incorrecto:**
/** @type {{ name: string }} */
let { name } = $props();

**✅ Correcto:**
interface Props { name: string }
let { name } = $props<Props>();
```
